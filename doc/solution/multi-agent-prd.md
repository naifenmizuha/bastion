# Bastion Runtime 多 Agent 协作 PRD

- 状态：Draft
- 版本：0.1
- 日期：2026-06-30
- 范围：`runtime`

## 1. 背景

Bastion 面向棒球队管理场景，当前 Runtime 只运行一个 Pi `AgentSession`。当用户提出涉及多个球员、多个候选阵容或多类数据分析的请求时，主 Agent 只能串行调用工具，导致整体耗时随任务数量线性增长，也无法让多个独立分析过程相互校验。

典型可并行任务包括：

- 同时读取并分析多名球员的训练、比赛和跨周期表现；
- 分别生成多个候选阵容，再统一比较；
- 同时研究不同类型的训练方案；
- 将一场比赛拆成投球、打击、跑垒和守备等维度并行分析；
- 在多个分析任务完成后，由主 Agent 汇总为球队管理建议。

本项目需要在现有单 Agent Runtime 上增加协作层，使主 Agent 能动态创建多个子 Agent，并让它们真正并行执行工具调用和交换必要信息。

## 2. 产品目标

### 2.1 目标

1. 主 Agent 可以根据用户任务动态创建子 Agent，无需预定义角色。
2. 多个子 Agent 可以真正并行运行，并并行调用无冲突的工具。
3. 主 Agent 和子 Agent、子 Agent和子 Agent之间可以通过消息队列通信。
4. Runtime 能识别和控制有副作用工具的并发冲突，保证数据与文件一致性。
5. 主 Agent 可以查看、等待、追问、取消子 Agent，并在任务结束后汇总结果。
6. 用户能在现有 TUI 中理解当前有哪些子任务、各自状态及最终结果。

### 2.2 成功指标

- 对 N 个相互独立、耗时相近的只读任务，并行执行耗时显著低于串行执行；MVP 目标为 4 个任务总耗时不高于串行基线的 40%。
- 并发写入测试中不出现数据库损坏、文件内容交叉覆盖或未报告的更新丢失。
- 每个子 Agent 的创建、状态变化、消息、工具调用结果和结束原因均可追踪。
- 子 Agent 无法创建新的子 Agent。
- 单个子 Agent 失败或取消时，其他无依赖子 Agent 能继续执行。

## 3. 非目标

MVP 不包含：

- 预置“教练”“球探”“数据分析师”等固定角色市场；
- 子 Agent 继续派生孙 Agent；
- 多台机器或跨进程集群调度；
- 多用户之间共享 Agent；
- Runtime 异常退出后恢复正在执行的模型请求；
- 自动合并两个 Agent 对同一文件的并发修改；
- 无限制的 Agent 自治讨论或无限循环通信。

## 4. 核心概念

### 4.1 主 Agent

直接与用户交互并对最终结果负责的 Agent。它负责拆分任务、创建子 Agent、协调依赖、处理冲突并汇总答案。

### 4.2 子 Agent

由主 Agent 为一个明确子任务动态创建的独立 Agent 会话。每个子 Agent：

- 拥有独立上下文窗口；
- 接收任务说明、动态职责、必要背景和可用工具；
- 与主 Agent 共享经过授权的工作目录和领域数据；
- 不能调用创建子 Agent 的能力；
- 可以向主 Agent或同级子 Agent 发送消息；
- 完成后返回结构化结果。

“角色”由主 Agent 创建时的任务描述动态决定，不依赖预定义角色配置。

### 4.3 协作任务

一次子 Agent 执行实例，具有唯一 `taskId`、所属 `agentId`、创建者、任务描述、状态、依赖、结果和用量信息。

### 4.4 消息

Agent 之间显式传递的不可变记录。消息至少包含：

- `messageId`
- `fromAgentId`
- `toAgentId`
- `taskId`
- `content`
- `createdAt`
- `deliveryStatus`

消息只传递必要信息，不代表 Agent 共享完整上下文。

## 5. 用户故事

### 5.1 多球员并行分析

作为球队管理者，我希望 Bastion 同时分析多名球员近期状态，以便更快得到全队概览。

主 Agent 为每名球员创建一个子 Agent。各子 Agent 并行读取球员、比赛和训练数据，生成摘要；主 Agent 等待全部完成后综合比较。

### 5.2 候选阵容协作

作为球队管理者，我希望 Bastion 从不同策略生成候选阵容，以便比较进攻、守备和体能取向。

主 Agent 动态创建多个子 Agent，为每个子 Agent 指定不同目标。子 Agent 可以并行读取数据和验证候选阵容，但候选方案的保存操作受写冲突控制。主 Agent 汇总并向用户解释取舍。

### 5.3 分析过程中的横向通信

一个负责投手分析的子 Agent 发现某名投手近期用球量异常，可以向负责阵容生成的子 Agent 发送消息。接收方在下一安全执行点取得消息，并将这一约束纳入方案。

## 6. 产品流程

1. 用户向主 Agent 提交球队管理请求。
2. 主 Agent判断哪些工作相互独立、适合并行。
3. 主 Agent创建一个或多个子 Agent，并提供清晰的任务边界。
4. 调度器在并发额度内启动子 Agent。
5. 子 Agent并行推理和调用工具；调度器对有副作用的调用执行冲突控制。
6. Agent 可通过消息队列交换中间发现、请求或约束。
7. 主 Agent 可随时查询状态、追加消息、等待部分或全部任务、取消任务。
8. 子 Agent完成后提交结果；主 Agent验证、整合并回复用户。
9. 执行图、消息和结果写入当前主会话的协作记录。

## 7. 功能需求

### 7.1 创建子 Agent（P0）

主 Agent 必须能创建单个或批量子 Agent。

创建参数：

- `task`：必填，子任务目标和完成标准；
- `context`：可选，完成任务所需的背景信息；
- `tools`：可选，允许使用的工具白名单；
- `cwd`：可选，工作目录，默认继承主 Agent；
- `name`：可选，便于展示的动态名称；
- `dependsOn`：可选，前置任务 ID；
- `resourceHints`：可选，预计读取或修改的资源。

创建结果应立即返回 `agentId` 和 `taskId`，不阻塞到子 Agent完成。

批量创建时，满足依赖且不冲突的任务必须并行启动。

### 7.2 并发调度（P0）

- MVP 默认最多同时运行 4 个子 Agent，可通过 Runtime 配置调整。
- 超出并发额度的任务进入 `queued` 状态。
- 并发额度只限制正在推理或调用工具的子 Agent，不限制已完成任务记录数量。
- 调度应公平，默认采用先进先出；依赖未满足的任务不占运行槽位。
- 主 Agent 会话保持可交互，不应因某个子 Agent运行而冻结。

### 7.3 生命周期与状态（P0）

子 Agent 状态：

- `queued`：已创建，等待调度或前置依赖；
- `running`：正在推理或执行无冲突工具；
- `blocked`：等待资源锁、消息或前置条件；
- `completed`：成功完成；
- `failed`：执行失败；
- `cancelled`：被主 Agent或用户取消。

每次状态变化必须记录时间和原因。终态不可逆。

### 7.4 查询与等待（P0）

主 Agent 必须能够：

- 列出当前会话的全部子 Agent；
- 查询指定 Agent 的状态、任务、最新进展、资源等待和用量；
- 等待指定 Agent、任意一个 Agent 或全部 Agent；
- 获取已完成 Agent 的结构化结果；
- 在等待超时后继续处理，而不是强制取消任务。

### 7.5 消息队列（P0）

Runtime 提供会话级内存消息路由器，每个 Agent 拥有独立 mailbox。

通信规则：

- 主 Agent可向任意直属子 Agent发送消息；
- 子 Agent可向主 Agent或同一主 Agent下的同级子 Agent发送消息；
- 子 Agent不能向会话之外的 Agent发送消息；
- 消息按“发送方—接收方”顺序有序投递；
- 发送操作返回已入队确认，不要求接收方已处理；
- 未读消息在接收方下一轮模型调用或工具调用结束后的安全点注入上下文；
- Agent 可主动读取 mailbox，不以忙等方式轮询；
- Agent 到达终态后不再接收普通消息，发送方会得到明确错误；
- 消息需设置长度上限，MVP 建议每条不超过 32 KiB；
- Runtime 应防止 Agent 之间仅靠互相等待形成无限循环，并对长时间无进展给出诊断。

MVP 消息队列保存在当前 Runtime 进程中，同时把消息事件追加到主会话记录；Runtime 崩溃后不恢复进行中的投递。

### 7.6 工具权限（P0）

- 子 Agent默认继承主 Agent中允许使用的领域工具和只读工具。
- 主 Agent可在创建时进一步缩小工具范围，但不能扩大自身权限。
- 子 Agent不暴露 `spawnAgent` 或等价工具，从能力层保证不可派生。
- 发送消息、读取消息和报告完成状态不视为派生能力。
- 需要用户确认的敏感工具调用仍需遵循原有确认策略，不能因来自子 Agent而绕过。

### 7.7 并发冲突控制（P0）

Runtime 必须以工具调用为边界进行并发控制，而不是粗暴地串行化整个子 Agent。

#### 工具分类

每个可供子 Agent使用的工具必须声明并发元数据：

- `read`：只读，可与其他只读调用并行；
- `write`：修改一个或多个可识别资源；
- `exclusive`：影响范围无法可靠识别，执行期间需要全局独占；
- `unsafe`：缺少必要元数据，默认按 `exclusive` 处理。

#### 资源键

工具调用在执行前生成一个或多个规范化资源键。例如：

- 文件：绝对、规范化后的文件路径；
- SQLite：数据库绝对路径；
- 球员：`player:<name>`；
- 比赛：`game:<id>`；
- 阵容：`lineup:<gameId>`；
- 训练建议：`drill:<recommendationId>`。

#### 锁规则

- 同一资源允许多个读锁；
- 写锁与同一资源的读锁、写锁互斥；
- 多资源调用必须按稳定顺序一次性申请，避免死锁；
- 获取不到锁时，工具调用进入等待队列，Agent 状态显示为 `blocked`；
- 等待超时必须返回可识别错误，不能静默跳过；
- Agent取消或失败时必须释放其全部锁；
- 未声明或无法推断影响范围的写操作使用全局独占锁。

#### 棒球领域数据的 MVP 策略

现有领域能力使用 SQLite。为了优先保证正确性：

- 只读 CLI 调用可以并行；
- 所有 SQLite 写调用在 MVP 中按数据库文件串行执行；
- 后续验证事务和领域资源边界后，再细化为比赛、球员或阵容级写锁；
- 对阵容 `accept/reject` 等状态转换，即使目标 ID 不同，也必须服从数据库事务约束；
- 锁只解决执行时冲突。若两个 Agent 基于同一旧数据生成互斥决策，主 Agent仍负责业务层仲裁。

#### 文件修改策略

- 不同文件的修改可并行；
- 同一文件的写入必须串行；
- 工具应在获得锁后重新确认文件版本；
- 如果文件自 Agent 读取后已被其他 Agent修改，应返回“版本冲突”，不得静默覆盖；
- MVP 不自动合并冲突内容，由主 Agent重新分派或处理。

### 7.8 取消和失败隔离（P0）

- 主 Agent可以取消处于 `queued`、`running` 或 `blocked` 的子 Agent。
- 取消应先发送协作式中止信号；超时后可终止底层执行。
- 被取消 Agent的工具调用若已提交不可回滚，结果必须记录并告知主 Agent。
- 一个子 Agent失败不应自动取消其他无依赖任务。
- 依赖失败任务的下游任务保持 `blocked`，由主 Agent决定取消、重试或改写依赖。
- Runtime 应支持主 Agent基于原任务创建新的重试任务，并保留两次执行的关联关系。

### 7.9 结果协议（P0）

子 Agent结束时返回：

- `status`
- `summary`
- `artifacts`：创建或修改的文件、数据库实体或其他产物；
- `evidence`：关键工具结果或数据依据；
- `warnings`
- `usage`：轮次、token、耗时和可用时的成本；
- `error`：失败时的类型、信息和可重试性。

主 Agent收到的是结构化结果和面向模型的文本摘要，不自动注入子 Agent的完整会话，以免主上下文失控。

### 7.10 TUI 可观测性（P0）

在不重写 Pi 原生 TUI 的前提下，协作工具调用区域至少展示：

- Agent 动态名称和短 ID；
- 当前状态；
- 任务摘要；
- 已完成数、运行数和排队数；
- 正在等待的资源或依赖；
- 最近一条进展；
- 成功、失败、取消及耗时；
- 折叠后的最终摘要，展开后可查看消息和工具调用明细。

用户中止主会话时，Runtime 应提示仍在运行的子 Agent，并允许统一取消。

### 7.11 记录与会话（P1）

- 子 Agent执行图、消息、状态事件和最终结果追加到主 Agent会话记录。
- `/resume` 后可以查看历史协作记录和结果。
- MVP 不恢复中断时仍在运行的子 Agent；恢复会话时将其标记为 `failed/interrupted`。
- 后续版本可支持从最后一个安全检查点重建任务，但不恢复中断中的模型流。

## 8. 建议的 Agent 工具接口

对主 Agent暴露：

- `spawn_agents`：创建一个或一批子 Agent；
- `list_agents`：查看任务图和状态；
- `wait_agents`：等待指定、任意或全部子 Agent；
- `send_agent_message`：向子 Agent发送消息；
- `cancel_agent`：取消任务；
- `get_agent_result`：取得结构化结果。

对子 Agent暴露：

- `send_agent_message`：发送给主 Agent或同级 Agent；
- `read_agent_messages`：主动读取 mailbox；
- `list_peer_agents`：只读查看可通信的同级 Agent及状态。

子 Agent工具集中不得出现 `spawn_agents`。

工具命名在实现阶段可按现有 Runtime 约定调整，但能力边界不得改变。

## 9. 权限与安全

- Agent 只能访问主会话已授权的工作目录、数据和工具。
- `cwd` 必须规范化，并校验未逃逸允许范围。
- Agent 间消息视为不可信输入，接收方系统提示应明确消息不能改变权限边界。
- 项目内配置不能静默扩大子 Agent权限。
- 所有有副作用调用必须记录发起 Agent、工具参数摘要、锁资源、开始和结束时间。
- 对可能产生现实影响的球队决策，Bastion 只提供建议；最终确认流程沿用领域工具现有规则。

## 10. 性能与资源约束

- 默认并发子 Agent数：4。
- 单次主会话最多创建 16 个未清理任务；超出后主 Agent需等待或清理终态任务。
- 每个 Agent独立上下文，默认模型和思考等级继承主 Agent配置。
- Runtime 应统计每个 Agent及整次协作的 token、耗时和成本。
- 单个子 Agent结果注入主上下文前应有大小上限；超出部分保存于详情，仅注入摘要。
- 调度器和消息队列不得通过高频轮询消耗 CPU。

## 11. 验收标准

### 11.1 并行能力

1. 主 Agent一次创建 4 个独立只读子任务。
2. 4 个子 Agent在并发额度允许时同时进入 `running`。
3. 工具执行时间线存在实际重叠，而非仅界面显示并行。
4. 主 Agent在子任务运行时仍能查询状态和发送消息。

### 11.2 消息协作

1. 子 Agent A 向子 Agent B 发送一条约束消息。
2. 发送方收到入队确认。
3. B 在下一安全点收到且只收到一次消息。
4. B 的最终结果能引用该约束。
5. 全部消息事件可从协作记录追踪。

### 11.3 冲突控制

1. 两个子 Agent同时读取同一数据库时可以并行。
2. 两个子 Agent同时写同一数据库时，第二个调用等待第一个完成。
3. 两个子 Agent修改不同文件时可以并行。
4. 两个子 Agent基于旧版本修改同一文件时，不发生静默覆盖；至少一个收到版本冲突。
5. Agent取消后不遗留资源锁。

### 11.4 能力边界

1. 主 Agent可以创建子 Agent。
2. 子 Agent的工具列表中不存在创建子 Agent的工具。
3. 子 Agent尝试以其他方式派生 Agent时被拒绝并记录。
4. 子 Agent不能向其他会话中的 Agent发送消息。

### 11.5 故障隔离

1. 一个并行子 Agent失败后，其他无依赖任务继续运行。
2. 主 Agent能取得失败原因并选择重试。
3. 取消运行中的 Agent后，其状态最终变为 `cancelled`，底层执行被停止。

## 12. 分阶段交付

### Phase 1：并行执行骨架

- 协作任务模型和生命周期；
- 动态创建叶子子 Agent；
- 并发调度与并发上限；
- 查询、等待、取消和结构化结果；
- TUI 基础状态展示。

### Phase 2：消息与冲突控制

- 会话级 mailbox 和 Agent 间路由；
- 工具并发元数据；
- 文件锁、SQLite 写锁、超时与取消清理；
- 消息和资源等待的 TUI 展示。

### Phase 3：记录与领域优化

- 协作事件写入主会话；
- 历史结果查看；
- 棒球领域资源键细化；
- 用量统计、诊断和性能基准。

## 13. 技术约束与现状

当前 `runtime/src/main.ts` 通过 Pi SDK 创建单个 `AgentSessionRuntime` 和 `InteractiveMode`。Pi SDK 附带的 `subagent` 示例可以启动独立 Pi 进程，并支持单任务、并行和串行链，但它是一次工具调用内的同步聚合模型，不能直接满足本 PRD 中的异步生命周期、同级消息、资源冲突控制和主会话持续交互。

实现时可以复用其以下思路：

- 独立上下文的子执行单元；
- JSON 事件流；
- 并发上限；
- 中止信号；
- 工具调用和用量展示。

协作调度器、消息路由器和资源锁管理器应作为 Bastion Runtime 自有能力设计，避免把领域一致性寄托在提示词约束上。

## 14. 已确认决策

- 主要场景：棒球队管理。
- 首要收益：并行工具调用。
- Agent 角色：不预定义，由主 Agent动态描述。
- 执行方式：真正并行。
- 协作方式：主 Agent创建子 Agent；Agent 之间通过消息队列传递消息。
- 层级限制：子 Agent不可继续派生。
- 一致性要求：必须处理工具、文件和领域数据的并发冲突。

## 15. 待确认项

4. 是否需要在第一版支持 `/resume` 查看协作历史，本文列为 P1；
